## 1. Objetivo y estrategia

El MVP actual funciona **100 % con mocks** (`packages/operativa/src/mocks/`): la lógica de liquidaciones, devengo de bonos, márgenes M1–M4 y CxC/CxP es real y está testada, pero los datos son ficticios. Supabase **SÍ es el destino**; la estrategia es por fases, sin saltos:

1. **Validar con mocks** (hecho): las reglas de negocio viven en funciones puras con tests; el modelo TS es la verdad del producto.
2. **Importar el Excel real** a una zona de staging para contrastar el modelo contra la realidad (pagos a trabajadores, gastos, cash flow).
3. **Diseñar el schema definitivo** = baseline #4 + A1 + A2 **más los huecos que detecta este documento** (matriz §2).
4. **Aplicar con autorización expresa** en un proyecto Supabase nuevo y limpio, siguiendo `APPLY_RUNBOOK.md` + checklists (§8). Hasta entonces, todo es borrador revisable.

La ventaja de este orden: cuando llegue el APPLY, el esquema ya habrá sido validado dos veces (contra la lógica del producto y contra los datos reales del Excel), y la base seguirá siendo **desechable** hasta que entre el primer dato real.

## 4. Datos que NUNCA se guardan (línea roja clínica)

Antifrágil OS es **administrativo-operativo**. En ninguna tabla, columna, nota ni JSON entra:

- diagnósticos, lesiones, evolución o notas clínicas;
- medicación, antecedentes, informes médicos;
- historia clínica en cualquier forma (ni adjunta en Storage).

La historia clínica vive en el sistema clínico externo. Lo único admisible, **en el futuro y previa decisión expresa**, es un identificador de referencia controlado (`external_clinical_record_id`) en `clinica_clientes` — hoy ni siquiera existe la columna (A2 lo difiere a propósito). `notas_admin` es administrativa; si algo clínico aparece en una petición, es fuera de alcance. Los checks del baseline (`post_bootstrap_checks.sql`, check 17) verifican la ausencia de rastro clínico en el esquema.

## 5. Migración mínima viable (orden de verticales)

Cada vertical entra solo cuando la anterior cuadra contra el mock y contra el Excel. Nada en paralelo.

**Fase A — operación diaria (lo que valida el negocio):**

1. **Clientes** → cargar `clinica_clientes` (A2 tal cual).
2. **Citas** → crear `clinica_citas` (nueva) sobre la agenda de A2; estados de liquidación incluidos.
3. **Cobros** → crear la tabla de cobros operativos enchufada a `cuenta_tesoreria`/`movimiento_caja` (A1).
4. **Bonos** → ventas + eventos de consumo + devengo; verificar invariante V1 con datos reales.

**Fase B — obligaciones y documentos:** 5. **Liquidaciones** → reglas por profesional, cálculo mensual, evidencia obligatoria, CxP derivadas. 6. **Facturas** → recibidas (circuito de #4 casi tal cual) y emitidas operativas (adaptar tabla; emisión oficial delegada, D1). 7. **Gastos** → tabla de gasto operativo + import del Excel de gastos.

**Fase C — visión financiera:** 8. **Tesorería** → arqueos, extractos bancarios, vista unificada; puente A1b. 9. **Rentabilidad** → márgenes M1–M4 como vistas sobre lo anterior (no persiste nada nuevo).

## 6. Seguridad

- **RLS obligatorio en todas las tablas** (regla suprema del repo). El baseline y A1/A2 ya lo traen, pero en v1 la política es permisiva (`authenticated FOR ALL`): antes de datos reales hay que **endurecer por rol** — direccion (todo), coordinacion (operación + liquidaciones), recepcion (citas/cobros/clientes, sin liquidaciones ni márgenes), profesional (su agenda y sus sesiones, nunca importes de otros). El mapa auth→rol sale de `clinica_usuarios` (adaptada, §2).
- **Libros append-only** (`factura_pagos`, `movimiento_caja`, `arqueo_caja`, y los nuevos: cobros, eventos de bono, devengos): cliente solo SELECT; escritura únicamente por RPC `SECURITY DEFINER` con actor derivado de `auth.jwt()`.
- **Sin fallback hardcodeado**: el fallback de ref + anon key legacy del cliente (`packages/supabase-client`) **ya fue eliminado**; la configuración por variables de entorno es **obligatoria** (`NEXT_PUBLIC_SUPABASE_*`, `VITE_SUPABASE_*`; `SERVICE_ROLE` solo en backend). Sin env → la app no arranca contra ningún proyecto por accidente.
- **Documentos privados**: buckets de Storage siempre `public=false`, SELECT solo authenticated, INSERT acotado por carpeta (patrón del bucket `facturas`); subida de documentos sensibles vía Edge Function con service role. Nóminas y evidencias de liquidación heredan este patrón, jamás un bucket público.
- **Rotación pendiente (R5)**: la **anon key legacy** que llegó a estar en el repo sigue pendiente de rotar. Debe rotarse antes de cualquier puesta en producción, y el proyecto nuevo genera claves nuevas — nunca se reutilizan las del legacy. Ninguna clave, URL de proyecto con secreto ni password entra en el repo: gestor de secretos.
- **Emails placeholder**: `configuracion_contabilidad` arranca en modo single-operator (3 emails iguales `.invalid`); poner los operadores reales es requisito previo a datos reales, no un detalle posterior.

## 7. Riesgos y rollback

| Riesgo                                                | Mitigación                                                                                                                                                             |
| ----------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Aplicar en el proyecto equivocado (legacy o Lidomare) | Checklist A del `SECURITY_CHECKLIST.md`: verificar project ref dos veces antes de ejecutar.                                                                            |
| SQL falla a mitad                                     | La base queda inválida: **no parchear a mano**; borrar el proyecto y recrear (es desechable hasta que haya datos reales — `ROLLBACK_NOTES.md`).                        |
| Doble conteo bono/sesión                              | El cobro del bono entra por la venta; la sesión de bono devenga pero **no** vuelve a cobrar (anti-doble-conteo ya modelado en el mock). Verificar V1 tras cada import. |
| Mezclar devengo y caja                                | Vistas separadas siempre; jamás se suman (doc 09 §4.4).                                                                                                                |
| Estados TS ≠ estados SQL (facturas, liquidaciones)    | Definir el mapeo en `packages/types` antes de recablear la UI; ningún estado se traduce "al vuelo" en componentes.                                                     |
| Roles heredados en RPC (`guille/javi/alicia`)         | Revisar/renombrar el mapeo de operadores antes de producción; hasta entonces el modo single-operator no protege nada.                                                  |
| Ya con datos reales                                   | Borrar el proyecto deja de ser opción: régimen de backups/PITR + migraciones aditivas; documentarlo al llegar ahí.                                                     |
